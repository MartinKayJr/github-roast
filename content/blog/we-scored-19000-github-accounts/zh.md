**摘要。** 人们普遍担心 GitHub 上存在伪造活动——模板化 Pull Request、买来的 star、互关刷粉网络——但关于这类造假到底有多少、长什么样，定量证据一直很少。我们使用一套确定性、开源的评分规则（不调用任何模型，相同输入必得相同分数）对 18,947 个公开 GitHub 账号评分，并对其中 3,444 个账号保留完整的原始指标快照作为深度样本。主要发现有四点：(i) 在本样本中造假很罕见——超过我们保守设定的刷量阈值的账号仅占 0.58%；(ii) 造假一旦出现就是极端形态，与正常行为呈双峰分离——被标记账号的模板化 PR 标题占比全部高于 50%（最高 97%），而总体中位数只有 7%；(iii) 低投入的贡献是常态而非可疑迹象——在有近期合并 PR 样本的开发者中，58% 至少有一个改动微小的外部 PR，而呈现规模化"以此为主"模式的只占 0.4%；(iv) 在被检测规则命中的账号中，占主导的形态是"空"而不是"假"——缺失型信号的出现率比操纵型信号高一个数量级。账号的中位数分数随年龄单调上升，从不满一年的 18 分升到十年以上的 86 分，这与"长期一致性是最难伪造的信号"这一假设相符。我们据此讨论反垃圾工具的设计启示，核心结论是：刷量是账号历史层面的模式属性，而不是单个贡献层面的事件属性。

## 1. 引言

对开发者可信度的评估正越来越依赖公开的 GitHub 活动：招聘流程筛查候选人主页，开源维护者判断陌生贡献者的 Pull Request 是否值得看，下游工具按表观影响力给账号排序。每一种用途都在制造伪造这些信号的动机。坊间不乏关于 star 交易、模板化 PR 攻势和互关刷粉的传闻；但对这类造假的发生频率及其统计形态的系统测量，并不多见。

数据集中的一个例子可以说明这一现象。某账号的合并 PR 记录在通常意义上足以证明其是强贡献者：大量被合并（而非仅提交）的 Pull Request，通过率接近满分。细看却发现，其近期 PR 标题中 97% 是几乎一模一样的模板变体，且多数指向同一个该账号并不拥有的热门仓库。没有任何一个 PR 单独看是异常的；异常只存在于整体模式的层面。这一观察——造假在单个事件层面看不出来，在聚合层面却一目了然——构成了本研究的出发点。

我们提出三个问题：

1. **比例。** 在公开 GitHub 账号中，伪造贡献活动有多常见？
2. **结构。** 造假发生时，它的统计特征与正常贡献行为差在哪里？
3. **构成。** 在被检测规则命中的账号中，主动造假与单纯的不活跃、缺乏原创工作各占多少？

为回答这些问题，我们使用一套确定性规则（[ghsphere](https://ghsphere.com)，评分核心以 AGPL 协议开源，见[代码仓库](https://github.com/MartinKayJr/github-roast)）对 18,947 个公开账号评分，并分析了保留完整原始指标快照（含 PR 级样本、仓库质量特征与活跃形态统计）的 3,444 个账号构成的深度样本。各图背后的聚合数据与本文一并发布（[data.json](/blog/we-scored-19000-github-accounts/data.json)）。

主要结论：在本样本中，造假远比公共讨论所暗示的少见；一旦出现就是极端形态，没有渐变地带；而且只靠模式层面的特征，就足以把它与真实活动近乎完全分开。

## 2. 数据与方法

### 2.1 评分规则

引擎实现了一套覆盖六个维度、合计 100 分的确定性规则，并对风险信号施加累加式扣分。引擎不调用任何模型，分数可完全由公开 GitHub 数据复现。ghsphere 网站、npm/PyPI SDK 与本文分析使用同一套代码路径。

| 维度 | 满分 | 奖励的信号 |
|---|---|---|
| 贡献质量 | 27 | 合并 PR 数（对数刻度）、通过率、issue 参与 |
| 生态影响力 | 20 | 对高星仓库的实质性 PR、深度维护参与 |
| 原创项目质量 | 18 | 按仓库实质内容加权的 star 数 |
| 活跃真实性 | 17 | 持续的近期活跃、活动类型多样性 |
| 账号成熟度 | 10 | 账号年龄、实际活跃年数 |
| 社区影响力 | 8 | 关注者数（对数刻度）、关注/被关注比是否合理 |

十二条确定性的风险信号规则负责扣分，包括 `templated_pr_flooding`（模板化 PR 灌水）、`trivial_pr_farming`（微小 PR 刷量）、`follow_farming`（互关刷粉）、`possible_star_inflation`（疑似 star 注水）等。具体阈值见代码仓库。在公开分数之外，引擎还计算一个 0–10 的内部垃圾/机器人可能性分数，用于保护榜单不被操纵；第 3.2 节首次公布其分布。除此之外，本分析不使用任何非公开数据。

**定义。** 若一个 Pull Request 改动不超过五行、且被合并进作者并不拥有的 200 star 以上仓库，我们称其为**微小 PR**。一个账号的**模板化标题占比**，指其近期 PR 标题中互为近似模板变体的比例。

### 2.2 样本构成与已知偏差

样本由两部分构成：(a) 通过 ghsphere 网站主动给自己账号评分的用户；(b) 从活跃开源组织收录的开发者。这一构成从两个方面限制了结论的适用范围。第一，样本是自选择的，天然偏向真实、活跃的开发者；因此下文报告的所有造假比例，都应理解为**一个已被过滤过的人群里的下限**，而非对 GitHub 全站的估计。第二，18,947 个已评分账号（其中 3,444 个含深度指标）足以刻画分布的形状，但相对 GitHub 总量微不足道；我们报告的是形状，不是普查。

## 3. 结果

### 3.1 分数分布

![19k 账号的最终分数分布](/blog/we-scored-19000-github-accounts/score-distribution.svg "图 1：最终分数分布，每 5 分一段（n = 18,947）。橙色标记 70 分以上段位。")

**图 1** 展示了最终分数的分布。中位数略高于 40 分；**48.6%** 的账号低于 40 分（规则将该档标注为低价值/疑似刷量），只有 **3.7%** 超过 90 分。人数最多的分数段是 0–5 分，由没有原创工作、没有合并 PR、没有持续活跃的账号构成。即使样本偏向活跃开发者，多数公开主页依然内容单薄。

作为参照：深度样本中位数账号拥有 **27 个关注者、34 个 star、20 个合并 PR**，账号年龄中位数为七年。与声誉相关的指标高度集中在头部：第 90 百分位是 1,275 关注者、约 5,900 star；第 99 百分位是 19,000 关注者、约 100,000 star。

### 3.2 刷量行为的比例与结构

![隐藏反刷分分布](/blog/we-scored-19000-github-accounts/spam-score.svg "图 2：内部 0–10 垃圾可能性分数的分布（n = 18,934）。77% 的账号得分恰好为 0。")

**图 2** 报告了 18,934 个计算过该分数的账号上，内部垃圾可能性分数的分布：

- **77%** 的账号得分恰好为 0——没有任何刷量迹象。
- **0.58%**（110 个账号）达到 3 分及以上，即我们判定为明确刷量的阈值。
- **12 个账号**达到 7 分及以上。

刷量账号之所以容易分辨，靠的不是数量，而是**集中度**。在近期至少有 10 个 PR 的 2,122 名开发者中，模板化标题占比的中位数是 **7%**——普通开发者也会适度地重复自己（"fix typo"、"bump deps"）。而所有因模板化灌水被标记的账号，占比全部高于 **50%**，最高达 **97%**；作为对比，全体人群的第 99 百分位也只有 72%。这个分布实际上是双峰的：在"正常重复"与"批量生成"之间不存在过渡地带。识别这类账号不需要机器学习模型，只需要在账号层面把多个 PR 放在一起统计。

### 3.3 微小 PR 是常态，靠微小 PR 刷量不是

在有近期合并 PR 样本的 2,558 名开发者中，**58%** 至少有一个符合 2.1 节定义的微小外部 PR。可见，向知名仓库提交小的文档或拼写修复，是开源参与的常规组成部分，对本身很强的贡献者也是如此。

相比之下，近期合并 PR **以微小改动为主**、且此类 PR 达到 10 个以上的开发者，只占样本的 **0.4%**（11 个账号）。

这两个数字之间的差距，对反滥用工具的设计有直接影响。任何单纯惩罚"小 PR"的启发式规则，都会为了抓住 11 个在模式层面本就显眼的账号，误伤那 58%——新人、文档贡献者、顺手帮忙的路过者。我们由此得出结论：刷量是账号贡献历史的**模式属性**（集中度、模板化、对单一目标的反复提交），而不是任何单个 diff 的**事件属性**；只根据单个事件就触发的检测器，量错了对象。

关于拒绝率的结果与此一致。在至少有 10 个已有明确结果（合并或关闭）PR 的 2,003 名开发者中，维护者拒绝率的中位数是 **2%**，第 90 百分位是 13%。也就是说，通过率几乎不携带判别信息——只要一个人的 PR 有被合并的记录，他提交的东西几乎都会被合并——只有极端的拒绝率（规则的标记阈值为 50% 以上）才有信号价值。

### 3.4 风险信号构成：缺的多，假的少

用当前引擎对全部 3,444 个深度样本账号重新计算，**17%** 命中至少一项风险信号。**图 3** 展示其构成。

![风险信号出现率](/blog/we-scored-19000-github-accounts/red-flags.svg "图 3：深度样本中各风险信号的出现率（n = 3,444）。缺失型信号比操纵型信号高一个数量级。")

出现率最高的三项信号——`ghost_profile`（幽灵主页，10%）、`no_original_work`（无原创工作，8.2%）、`mostly_forks`（几乎全是 fork，7.2%）——描述的是内容的缺失，而不是造假。主动造假类信号低一个数量级：`templated_pr_flooding` 为 0.5%，`trivial_pr_farming` 为 0.3%，`follow_farming` 为 0.1%。star 注水（star 很高而 fork 与 issue 接近于零）在本样本中约等于零；不过自选择很可能压低了这个数字——买过 star 的账号，大概不会主动来接受评分。

对于建立在 GitHub 数据之上的信任工具，这个构成意味着两种性质不同的失败情形，需要不同的检测器和差别很大的容错标准：常见而代价低的一种（"这里什么都没有"），与罕见而代价高的一种（"这里有人造了假"）。

### 3.5 账号年龄：造不出来的信号

![账号年龄与中位数分数](/blog/we-scored-19000-github-accounts/age-vs-score.svg "图 4：按账号年龄划分的中位数最终分数。该关系在所有年龄段上均单调。")

中位数分数随账号年龄单调上升，从**不满一年账号的 18 分升至十年以上账号的 86 分**，中间没有任何回落（**图 4**）。幸存者偏差对此有贡献——出现在样本里的老账号，是仍在被使用的老账号——但这个效应的方向本身就说明问题。长期一致性的每一个组成部分（多年的持续活跃、star 自然积累的老仓库、跨越多个版本的贡献历史）恰恰是造假无法压缩的：star 和关注者几个小时就能买到，而一个 2015 年注册、活跃了九年的账号，没有办法在 2026 年凭空造出来。这种不对称性支持了规则中对账号成熟度与活跃年限的加权，也与我们观察到的现象一致：被标记的灌水账号集中在年轻账号里。

顺带一提，高分账号（最终分数 ≥ 60）的主语言构成符合大家熟悉的生态趋势：**TypeScript（520）、Python（460）、JavaScript（395）** 领先，Rust（225）高于 Go（189）、C（184）与 Java（159）（**图 5**）。

## 4. 讨论

上述结果指向贡献真实性工具的三条设计原则。

**看模式，不看单个事件。** 3.2 节的双峰分离与 3.3 节 58% 对 0.4% 的差距表明：单个贡献几乎不携带造假信息，而账号级的聚合统计能干脆地分开两类人。基于单个事件的启发式规则，会把误伤最大程度地落在开源项目最需要留住的人——新人和低频帮助者——身上。

**分清"空"和"假"。** 没通过检测的账号多数是空的，不是恶意的（3.4 节）。把两者混为一谈，会夸大表面上的造假率，也会把审查精力用错地方。

**给时间以权重。** 在所考察的信号中，长期一致性是唯一无法靠花钱缩短获取时间的信号（3.5 节），因此是任何可信度评分的天然锚点。

依据这些原则，我们正在构建一个帮助维护者初筛垃圾 PR 的 GitHub App，把 PR 级特征（diff 大小、模板相似度）与作者级历史结合起来。鉴于 3.3 节记录的误伤风险，这个工具不会自动关闭任何 PR，而是只呈现证据，把决定留给人。欢迎受到刷量困扰的仓库维护者向我们提供垃圾 PR 的样例。

## 5. 局限性

- **自选择样本。** 网站用户加活跃开源组织成员，偏向真实活跃。GitHub 全站的造假率很可能高于本文报告值；本文的百分比是已过滤人群内的下限。
- **规模。** 18,947 个已评分账号，3,444 个含深度指标。分布形状在多次重跑中保持稳定，但样本不能代表 GitHub 全体。
- **只做聚合报告。** 所有模式均以聚合形式报告，不指认任何具体账号。信号阈值全部公开，以便维护者结合上下文自行判断。
- **引擎版本。** 风险信号出现率以当前评分器对每个账号最新原始快照重新计算；存储的垃圾分数反映扫描当时的引擎版本。二者均包含在[公开聚合数据](/blog/we-scored-19000-github-accounts/data.json)中。

## 6. 可复现性

全部评分逻辑均为确定性且以 AGPL 协议开源：[github.com/MartinKayJr/github-roast](https://github.com/MartinKayJr/github-roast)。同一引擎通过 `npm install ghfind` 与 `pip install ghfind` 分发，既可调用公开 API（[OpenAPI 规范](https://ghsphere.com/openapi.json)），也可携带用户自己的 GitHub token 完全在本地运行。本文所有图表背后的聚合统计见 [data.json](/blog/we-scored-19000-github-accounts/data.json)。

*单个账号可在 [ghsphere.com](https://ghsphere.com) 进行评分。*
